印闪科技

  • cas问题

    • 产生的原因
      • 多个线程对共享资源的修改有可能会有导致数据不一致
      • 举例：比如一个全局属性int i = 1,线程A和线程B同时进行++1的操作，线程A将i在工作内存中修改完成后写回主内存（值为2），此时线程B也修改成2在写回主内存（值为2）。
      • 结果：两个线程都对i进行+1，但是结果确实2
      • 解决：
        • 1 synchronized关键字，加锁，太重了
        • cas，atomicInterger类解决，引用类型用atomicReference
    • 解决
      • atomicReference

  • 什么时候用编程式注解，声明式注解

    • 编程式事务
      • 事务粒度更细
    • 声明式事务
      • 非侵入式开发方式，事务粒度在一个方法范围

  • Spring事务情况下不生效

    • 数据库引擎不支持事务,例如mysql，innoDB支持, MyISam不支持

    • 本类调用本类的transaction修饰的方法

      • 举例：serviceA中的methodA（没有声明事务）调用本类中的methodB（声明了事务），methodB事务不生效

      • 原因

        • 被@transactional修饰的serviceA类 spring会给它创建一个代理类，当controller调用methodA时，由于没有被@transactional修饰，不用调用代理类的增强方法，methodA方法调用本类的methodB自然也不会走到代理类去

      • 解决方式

        • 想办法拿到serviceA的代理类，通过代理类调用methodB

        • 代码

          • public class ServiceLogProxyUtil {

    public static Object getProxy(Class<?> clazz, Object target) {
        return Proxy.newProxyInstance(Thread.currentThread().getContextClassLoader(), new Class[]{clazz}, new InvocationHandler() {
            @Override
            public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
                Object ret = method.invoke(target, args);
                System.out.println("after execute method:" + method.getName());
                return ret;
            }
        });
    }

}

TestTransactionalService proxy = (TestTransactionalService) ServiceLogProxyUtil.getProxy(this.getClass(), testTransactionalService);

    • private修饰的方法

      • 只有public方法才会被代理类代理到，所以事务方法必须是public

    • 默认是RuntimeException才会回滚

    • 事务入口和业务在同一个线程里
